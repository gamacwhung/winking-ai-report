第一部分：內部說明 — 團隊 AI 使用現況總覽

本次分析共涵蓋 4 位成員、6 段對話、共 28 輪 user 訊息，對方團隊使用 Cursor IDE + Unity MCP 進行 Unity C# 遊戲專案開發（模擬經營類遊戲，含 GVE 卡牌系統、商店系統、劇情系統、天氣/晝夜系統等模組）。

整體使用量偏保守：4 人在一天內僅產生 6 段對話（28 輪），其中有成員在對話中直接詢問「Token 會不會用完」「是否免費」，顯示團隊對 Token 額度與計費方式尚不清楚，可能影響使用意願與深度。

亮點行為：

  • 不讓 AI 直接修改程式碼 — 全程以 AI 作為「分析助手」，自己掌控修改決策，確保程式碼品質

  • 會從單一類別逐步要求整理「所有關聯類別的完整調用關係」
  • 持續深入追問同一個系統的不同面向

💡 給劉思源的建議：

  • 你的「只分析不讓 AI 改 code」風格是品質最穩的方式，非常值得肯定
  • 可以嘗試讓 AI 產出修改草稿但不自動 Apply（Cursor 有「不自動套用」的模式），你在 diff 上逐行審查後再手動接受，這樣既保留人工把關又能節省手動改寫的時間
  • 你的「截圖讓 AI 辨識腳本」方式很有創意，但精度有限；建議搭配 @檔名 語法，AI 能讀到完整程式碼，分析結果會更精確
  • 「請整理調用關係」的模式很好，可以再進一步要求 AI 輸出 Mermaid 圖，貼到文件裡變成可視化架構圖

第四部分：整體建議 — 接下來怎麼做

🎯 核心判斷

這個團隊不是不會用 AI，而是還不確定能不能放心用。
從對話記錄來看，4 位成員都能用中文自然地跟 AI 互動、描述需求也清楚，
但整體使用量偏低（一天 28 輪），且有成員直接問「Token 會不會用完」。
解除 Token 不確定性後，使用深度和頻率預期會自然提升。

建議方向一：釐清 Token 與計費限制

問題：團隊成員不確定 Token 額度和計費方式，導致使用保守、不敢讓 AI 做更多事。

你Token會用完嗎
你目前的計費是免費嗎

建議做法：

  • 向團隊明確說明目前的 Token 額度、計費方式、用量上限
  • 讓大家知道「正常開發使用不需要省」，降低心理門檻
  • 如果有用量限制，告知大概能支撐多少對話量，避免成員做到一半擔心額度中斷

建議方向二：場景化的 Prompt 教學

問題：團隊目前以「分析」為主（佔 60%），還沒充分發揮 AI 在修改、設計、Review 上的潛力。

建議做法：針對他們實際的遊戲開發場景，提供具體的 prompt 範本。以下是從他們對話中提取的實際案例和建議進階寫法：

建議做法：

  • 定期查看 agent-transcripts，了解使用狀況的變化（目前已在進行中）
  • 觀察使用量和使用模式的演進 — 特別關注從「純分析」到「分析+修改」的比例是否提升
  • 發現好的使用範例時，可以在團隊內分享，形成正向循環

📌 一句話總結

團隊的 AI 使用習慣與表達能力都沒問題，瓶頸在 Token 限制的不確定性。
先讓他們安心用，再搭配場景化教學提升效率，效果會最明顯。

第五部分：觀察到的技術風險（供團隊參考）

在分析過程中，AI 反覆識別出幾個共同的程式碼品質議題：

風險等級問題描述涉及模組
高 事件訂閱/取消訂閱不對稱 — 多處 += 沒有對應的 -=，或在錯誤的生命週期解除 UIGVECardOverviewMain, UICard
高 批次 RPC 回調造成多次 UI 刷新 — 銷毀卡牌等操作逐張發送，每次回調都重建 UI UIGVECardOverviewMain
中 硬編碼數值 — 等級上限 3、plotId == 0 代表關閉等 magic number UIGVEStrengthen, PlotManager
中 陣列長度未防護 — 多處假設陣列長度一致，缺少邊界檢查 PlotTalkPlayer, PlotOptionPlayer, BuildingManager.Floor
中 Dispose / 生命週期管理不完整 — 子系統未正確釋放 PlotManager, WeatherManager
低 Shader 全域參數命名一致性 — _TimeOfDay 在不同 Shader 中需統一 Weather/DayNight 系統

以上技術風險都是 AI 工具能很有效地協助識別與修復的領域，建議團隊優先處理高風險項目。
