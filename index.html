第二部分：各成員詳細使用分析

劉思源 — 最深入的使用者

使用模式：從編譯錯誤排查 → 程式碼架構分析 → 系統設計諮詢 → Shader/渲染技術方案，持續追問、逐層深入。

亮點行為：

  • 會主動糾正 AI 的錯誤 — AI 把 SyncWeatherState 綁到事件上時，他指出「這樣不就不能平滑」，說明他清楚系統設計意圖
  • 會提供領域知識作為約束條件 — 例如提醒「IsIntoStore 是正在店中活動的意思，門外也可能是 true」、「客人不適合直接 SetColor，因為其他功能也會用」
  • 會延伸探索 — 從分析單一類別到要求整理所有關聯類別的完整調用關係

涵蓋的 AI 使用場景：

  1. 解決 CS0246 編譯錯誤
  2. 分析 UI 腳本及完整調用鏈（UIGVECardOverviewMain 系列）
  3. 分析劇情系統（PlotManager / PlotSystem / 各種 Player）
  4. 天氣/晝夜系統設計與實作諮詢
  5. 搜尋遊戲邏輯（客人/店員進出門判斷）
  6. Shader 渲染問題除錯（PostprocessingMode）
  7. IDE 操作教學（Cursor 的 Go to Definition）
  8. 一般性問題（Token 限制、計費）

紀閎耀

使用模式：直接請 AI 分析指定腳本、要求「整理可以優化的部分」，屬於單次任務型。

具體場景：

  • 請 AI 分析 PlotTalkPlayer.cs 的 Play 函數並新增自動播放功能
  • 遇到工作區未同步的問題，AI 找不到檔案，對話中斷

趙健智

使用模式：僅一次短對話，給出一個簡單的程式修改需求。

具體場景：

  • 希望在 PlotTalkPlayer.cs 的 Play 函數新增 bool 參數（自動播放下一句）
  • 同樣因工作區路徑問題，AI 無法定位檔案

高偉傑 — 最完整的「分析→修改」流程

使用模式：善用 @ 語法附件檔案、從分析到修改的完整流程（分析 → 選方案 → 執行修改）。

具體場景：

  1. 分析 GamePass 系統所有腳本（Manager / System / Data / Extension）
  2. 修復 UIGamePassPanel 跨平台編譯錯誤（#if UNITY_ANDROID 在 Win 平台的 fallback）
  3. 分析 BuildingManager.Floor.cs 與 BuildingFloor.cs，並讓 AI 做防呆重構（A 方案）
  4. 分析 UIGVECardOverviewMain.cs 與 UIGVEStrengthen.cs

共同特點

第四部分：給唯晶團隊的建議報告

A. 環境設定（最優先）

建議 1：確保工作區路徑正確且檔案已同步

部分成員使用時 AI 找不到指定腳本，通常是因為 Cursor 開啟的工作區路徑與實際專案路徑不一致，或檔案尚未加入版控。

  • 確認 Cursor 開啟的資料夾是專案根目錄（例如 GameProject/Model.Unity）
  • 使用前確認目標檔案已存在於工作區中
  • 若使用 Perforce 或其他版控，先 sync 最新版本

建議 2：Unity MCP 連線確認

劉思源的對話中確認了 Unity MCP 可以正常連線。建議所有成員在開始使用前，都先確認 Unity Editor 是否正確連接（可以問 AI「你現在有跟 Unity 連在一起嗎？」）。

B. 使用技巧（提升效率）

建議 3：善用 @ 語法指定檔案範圍

高偉傑的使用方式是很好的範例 — 用 @UIGVECardOverviewMain.cs (1-382) 明確指定要分析的檔案和行數。

✅ 好的用法：@BuildingManager.Floor.cs (1-213) 請幫我分析這個腳本
⚠️ 一般用法：請分析 BuildingManager.Floor.cs
❌ 較差用法：幫我看一下那個建築樓層的程式

建議 4：從「分析」進階到「修改」的完整工作流

目前團隊大部分使用停留在「請 AI 分析」的階段。建議嘗試以下流程：

  1. 分析 →「請分析 XXX.cs 的職責與風險點」
  2. 確認 → 確認 AI 理解正確，如有誤差要糾正（像劉思源做的那樣）
  3. 方案選擇 →「請給我 A/B 兩種修法」
  4. 執行修改 →「請幫我做 A 方案的修改」
  5. 驗證 → 確認修改後編譯通過、邏輯正確

高偉傑在 BuildingManager.Floor.cs 的使用就是完美範例。

建議 5：給 AI 領域知識作為前提條件

劉思源在搜尋「進出門判斷」時，先說明了 IsIntoStore 的真正語意，讓 AI 能更精準地找到正確的程式碼。建議在提問時主動提供：

  • 專案特有術語的解釋
  • 已知的約束條件
  • 不希望被改動的部分
✅ 好的範例：
「尋找客人進出門的判斷。注意 IsIntoStore 是正在店中活動的意思，
 店外也有商品，所以在門外 IsIntoStore 也可能是 true。」

建議 6：批次分析多個相關檔案

劉思源展示了很好的模式 — 從一個檔案開始，逐步展開到關聯類別。建議一次列出相關檔案：

✅ 好的做法：
「請分析 PlotOptionPlayer.cs、PlotTalkPlayer.cs、PlotTitlePlayer.cs、
 PlotVideoPlayer.cs、UIPlotPlayer.cs，整理它們的職責分工與調用關係」

C. 進階使用（提升價值）

建議 7：讓 AI 幫助建立程式碼文件

可以利用 AI 分析結果，直接產出：類別職責說明文件、系統間調用關係圖（Mermaid 格式）、新人上手指南。

建議 8：利用 AI 做 Code Review

除了「請分析」之外，可以更具體地請 AI 檢查：

  • 「這段程式有沒有記憶體洩漏的風險？」
  • 「這個事件訂閱/取消訂閱是否對稱？」
  • 「這段 DOTween 在物件 Disable 時會不會殘留？」

建議 9：新功能開發前先做技術方案討論

劉思源在晝夜系統的討論展示了 AI 在設計階段的價值。建議團隊在開發新功能前，先跟 AI 討論技術方案，獲得多種實作思路與 trade-off 分析。

建議 10：避免的使用方式

  • 不要詢問非技術問題（「Token 會不會用完」「是否免費」）— 消耗上下文空間
  • 不要在工作區未同步的情況下讓 AI 修改程式碼
  • 不要盲目接受 AI 修改 — 應如劉思源所做，主動驗證 AI 輸出是否符合系統設計意圖

第五部分：觀察到的技術風險（供團隊參考）

在分析過程中，AI 反覆識別出幾個共同的程式碼品質議題：

風險等級問題描述涉及模組
高 事件訂閱/取消訂閱不對稱 — 多處 += 沒有對應的 -=，或在錯誤的生命週期解除 UIGVECardOverviewMain, UICard
高 批次 RPC 回調造成多次 UI 刷新 — 銷毀卡牌等操作逐張發送，每次回調都重建 UI UIGVECardOverviewMain
中 硬編碼數值 — 等級上限 3、plotId == 0 代表關閉等 magic number UIGVEStrengthen, PlotManager
中 陣列長度未防護 — 多處假設陣列長度一致，缺少邊界檢查 PlotTalkPlayer, PlotOptionPlayer, BuildingManager.Floor
中 Dispose / 生命週期管理不完整 — 子系統未正確釋放 PlotManager, WeatherManager
低 Shader 全域參數命名一致性 — _TimeOfDay 在不同 Shader 中需統一 Weather/DayNight 系統

以上技術風險都是 AI 工具能很有效地協助識別與修復的領域，建議團隊優先處理高風險項目。
